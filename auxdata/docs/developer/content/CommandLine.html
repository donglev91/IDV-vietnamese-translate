[header {Command Line Arguments}] If you run the IDV standalone you can print out the following usage message: 
java ucar.unidata.idv.DefaultIdv -help
Usage: IntegratedDataViewer 
	-help (this message)
	-properties <property file>
	-nodefault (Don't read in the default bundle file)
	-default <.xidv file>
	-nopreferences (Don't read in the user preferences)
	-nogui (Don't show the main window gui)
	-chooser (show the data chooser on start up) 
	-sitepath <url path to find site resources>
	-userpath <user directory to use>
	-display <paramater> <display>
	-data <data source> 
        (and any .xidv bundle files or .jnlp embedded bundle files)
Each of these command line arguments can also be defined in the jnlp webstart files (See index.html).

You can define any number of property files with the -properties argument. Each successive property file will overwrite the previous values for properties defined in the property file.

The -data argument allows you to specify the url pointing to a data source. This could be a local file, a file on a web server or a dods or adde server url.

The -userpath allows you to overwrite the default location where the user's idv preferences are kept. Normally this is in a directory: <user home>/.metapps/<Idv application class name>.

-nodefault says to not load in the default bundle.

-default <.xidv file> says to not load in the default bundle but do load in the given bundle.xidv file.

The -sitepath argument allows you to define where site specific resources are kept. This can be on the local file system or it can be a base url. This value is used in the properties file (see example.properties) to substitute the %SITEPATH% macro.

The -display argument allows you to automatically create a display for the given parameter for any data sources that are loaded in. The display name should be one of those contained in the controls.xml file (/ucar/unidata/idv/resources/controls.xml).

You can also specify any number of .xidv bundle files or .jnlp embedded bundle files directly on the command line. These can be local files or can be urls.

Processing your own arguments

If your application needs to read its own command line arguments you need to create your own ArgsManager class and create it by overriding the: doMakeArgsManager method.

Your ArgsManager class needs to overwrite the base class method: 

protected int parseArg (String arg, String \[] args, int idx)
See the documentation of this method in [dg::class ucar.unidata.idv.ArgsManager]
